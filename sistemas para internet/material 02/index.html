Anatomia: A URL (Identidade)

Toda requisição começa com um endereço único para o recurso.

https://api.meta.edu.br/v1/usuarios?ativo=true
  • 1
    Host: Onde o servidor mora (api.meta.edu.br).
  • 2
    Path: O caminho para o "armário" de dados (/usuarios).
  • 3
    Query Params: Filtros para a busca (?ativo=true).

Anatomia: Verbos e Headers

⚡ Verbo (Método)

Define a intenção.

Exemplo: "Eu quero BUSCAR algo" ou "Eu quero DELETAR algo".

🔖 Headers (Cabeçalhos)

Metadados da mensagem.

Contém tokens de autenticação, tipo de arquivo (JSON) e cookies.

Anatomia: O Body (Corpo)

Onde a carga pesada é carregada. Usado principalmente em métodos de criação e atualização.

// Exemplo de Body em JSON
{
  "nome": "João Silva",
  "email": "joao@meta.edu.br",
  "projeto": "Sistema de Gestão Escolar"
}
Dica de Arquiteto: Nunca envie senhas ou dados sensíveis via URL (Query Params). Use sempre o Body com HTTPS!

2. Teoria: Verbos HTTP e CRUD

No desenvolvimento de APIs, mapeamos os métodos HTTP para as operações fundamentais de banco de dados (CRUD).

Método Ação CRUD Exemplo no TCC
GET Read Listar meus usuários.
POST Create Cadastrar novo registro.

Verbos HTTP e CRUD (II)

Método Ação CRUD Exemplo no TCC
PUT / PATCH Update Editar perfil do usuário.
DELETE Delete Remover um registro.

Diferença técnica: PUT substitui o recurso todo. PATCH altera apenas partes específicas (ex: apenas o e-mail).

Mergulho no GET

O GET deve ser Idempotente e Seguro.

  • Não altera o estado do servidor (apenas lê).
  • Pode ser "cacheado" pelo navegador.
  • Usado para buscas, listagens e detalhes.

GET /api/v1/relatorios/45

Mergulho no POST

O POST cria vida no seu sistema.

  • Altera o estado do servidor (insere no banco).
  • Exige um Body com os dados.
  • Não é cacheável por padrão.

POST /api/v1/auth/login

Mergulho no Ciclo de Edição

PUT/PATCH

Envia o ID na URL e os dados novos no Body.

DELETE

Envia o ID na URL. O servidor remove o recurso permanentemente.

3. API Design: Rotas Semânticas

Suas URLs devem representar substantivos (Recursos), não verbos.

❌ Design Amador

  • /get_usuarios
  • /salvarDadosNoBanco
  • /delete-user?id=10
  • /user/edit/5

✅ Design Profissional

  • GET /usuarios
  • POST /usuarios
  • DELETE /usuarios/10
  • PATCH /usuarios/5

A Estrutura Padrão (Best Practices)

/api/v1/{recurso}/{id}

/api: Identifica que é uma rota de dados, não de páginas HTML.

/v1: Versionamento. Permite criar a v2 no futuro sem quebrar o app antigo.

/{recurso}: Sempre no plural (ex: /produtos, /livros).

Status Codes: Como o Servidor fala?

O código de status diz ao front-end o que aconteceu, sem precisar ler o texto da resposta.

2xx
Sucesso: Tudo deu certo.
4xx
Erro do Cliente: Você enviou algo errado.
5xx
Erro do Servidor: O back-end "explodiu".

Meus Status Codes Favoritos

200 OK (Sucesso)
201 Created (Criado)
400 Bad Request
401 Unauthorized
404 Not Found
500 Server Error

Box: Melhores Práticas de Mercado

The Architect's Handbook

  • ⚡ Use Kebab-case nas URLs (/meus-pedidos).
  • ⚡ Nunca use extensões de arquivos (.php, .html) na rota.
  • ⚡ Sempre retorne JSON, nunca texto puro ou HTML.

5. Desafio: 'O Garçom da API'

Como "Garçom" (API), qual Status Code você entrega nestas situações?

O cliente pediu um prato que não existe no menu.
O prato foi preparado com sucesso e entregue.
O cliente tentou entrar na cozinha (área restrita).

Quiz de Verbos HTTP

"Trocar a senha do usuário logado"

Cenário de TCC: Autenticação

Se o seu TCC tem login, como é a rota?

POST /api/v1/auth/login
Payload: { email, senha }

RES: 200 OK + Token JWT

Observe que usamos POST porque estamos "Criando" uma sessão (Token).

Cenário de TCC: Filtros

Se o seu TCC lista itens (ex: agendamentos), use Query Params:

GET /api/v1/agendamentos?data=2024-12-02

Isso permite que o front-end carregue apenas o necessário, economizando banda.

Evite estes Erros Críticos!

  • 1. Erro 200 Fake: Retornar status 200 com uma mensagem de erro no JSON ("erro": "senha invalida"). Se houve erro, o status deve ser 4xx!
  • 2. Verbos errados: Usar GET para deletar registros (pode ser executado por crawlers do Google!).
