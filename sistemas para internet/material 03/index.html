2. Teoria: O Padrão MVC

O MVC é um padrão de arquitetura que divide a aplicação em três componentes principais:

M Model
V View
C Controller

Model (Modelo)

"O Coração dos Dados"

  • ● Lida diretamente com o Banco de Dados.
  • ● Contém as Regras de Negócio (ex: "o saldo não pode ser negativo").
  • ● Define a estrutura dos objetos (Esquemas).
// Exemplo: usuarioModel.js
const user = { nome: String, email: String };

View (Visão)

"O que o Usuário vê"

  • ● Interface do Usuário (HTML, CSS, JSON).
  • ● Não deve conter lógica de banco de dados.
  • ● Apenas exibe os dados fornecidos pelo Controller.

Controller (Controlador)

"O Maestro do Sistema"

  • ● Intermediário entre View e Model.
  • ● Recebe a Requisição HTTP.
  • ● Chama o Model, processa os dados e envia para a View.

O Fluxo da Informação

VIEW (Cliente pede)
↓
CONTROLLER (Decide)
↓
MODEL (Busca Dados)
↑
CONTROLLER (Prepara)
↓
VIEW (Usuário recebe)

Vantagens Arquiteturais

Manutenibilidade

Mudar o banco de dados afeta apenas a pasta /models.

Flexibilidade

Mudar o layout (Tailwind para Bootstrap) afeta apenas /views.

3. Estruturação das Pastas (Node.js)

Crie a seguinte hierarquia no diretório /src do seu projeto:

  • 📁 src
    • 📁 config // Conexão BD
    • 📁 controllers // Lógica das rotas
    • 📁 models // Entidades e DB
    • 📁 views // Front-end
    • 📁 routes // Definição das URLs
    • 📄 app.js // Ponto de entrada

Pasta /routes

Aqui ficam os arquivos que você planejou na aula anterior (12/02).

// routes/userRoutes.js
router.get('/usuarios', userController.listar);

Cada arquivo de rota deve delegar a execução para um controlador correspondente.

Pasta /controllers

Onde a mágica do processamento acontece.

// controllers/userController.js
exports.listar = (req, res) => { ... };

O controlador não pergunta ao banco de dados diretamente; ele pede ao Model.

Pasta /models

Contém as definições de como os dados são salvos.

// models/userModel.js
const UserSchema = new Schema({ ... });

Pasta /config

Segurança e Infraestrutura.

Guarde aqui as credenciais do banco de dados e chaves de API secretas. Utilize arquivos .env para não expor senhas no GitHub!

Template Inicial de Controller

Utilize este template para criar seus controladores iniciais no TCC:

// src/controllers/genericController.js

exports.getAll = async (req, res) => {
    try {
        // 1. Chamar o Model para buscar dados
        // const data = await Model.find();
        
        // 2. Responder com sucesso
        res.status(200).json({
            status: "success",
            results: 1,
            data: { message: "Dados recuperados com sucesso" }
        });
    } catch (err) {
        // 3. Responder erro (Status 400 ou 500)
        res.status(400).json({ status: "fail", message: err });
    }
};

Quiz de Manutenibilidade

"Se precisarmos mudar o banco de dados de MySQL para MongoDB, qual camada será a mais afetada?"

Simulador de Banca

Pergunta da Banca: "Onde estão as regras de acesso do seu sistema?"

Desenvolvedor: "Estão concentradas nos Controllers, que validam a requisição antes de solicitar ao Model."

Resposta perfeita. Demonstra que você sabe quem controla o fluxo.

Referência: Projeto Real

Em grandes empresas (Google, Uber, Nubank), a estrutura é similar, porém com mais camadas:

  • /services (Lógica de negócio complexa)
  • /middlewares (Segurança e Validação)
  • /utils (Funções auxiliares)
