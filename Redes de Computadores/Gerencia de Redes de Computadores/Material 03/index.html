O Administrador como Detetive

Esqueça a sorte. A infraestrutura moderna exige uma abordagem científica para identificar a causa raiz de falhas antes que elas custem milhões.

Nível: Engenheiro de Confiabilidade (SRE)

Chega de "Tentativa e Erro"

Muitos técnicos juniores começam a trocar cabos, reiniciar servidores e alterar configurações aleatoriamente quando surge um problema. Isso é **perigoso**.

  • Dificulta o Isolamento: Se você muda três coisas e o problema some, você não sabe qual era a causa raiz.
  • Cria Efeitos Colaterais: Uma mudança "chutada" pode gerar um novo incidente mais grave.
  • Inconsistência: O problema pode "sumir" e voltar pior em uma hora.

"Se você tem apenas um martelo, todos os problemas parecem pregos. Metodologia é ter a caixa de ferramentas completa."

MTTR: A Métrica de Sucesso

O objetivo de um gestor de redes não é apenas "consertar", mas reduzir o **MTTR (Mean Time To Repair)**.

Definição MTTR

Média de tempo entre a descoberta da falha e a resolução completa.

EficiênciaMeta SRE

Reduzir o MTTR significa ter processos tão claros que o diagnóstico flui sem interrupções de raciocínio.

A Metodologia de 5 Passos

01 Observação e Coleta
02 Isolamento de Escopo
03 Análise de Causa Raiz (RCA)
04 Plano de Ação e Correção
05 Verificação e Documentação

Fase 01: O Que Está Acontecendo?

"A rede está lenta."

Este sintoma é vago. No passo 01, o objetivo é extrair o máximo de fatos técnicos:

Perguntas Chave

  • • Qual aplicação específica está lenta?
  • • Quando começou? (Data/Hora exata)
  • • É intermitente ou contínuo?

Ferramentas

  • • Ping / Traceroute
  • • Zabbix Dashboards
  • • Relatórios de Usuários

Fase 02: Definindo o Escopo

A Arte de Isolar

Isolar é definir quem **não** está sendo afetado. Isso economiza 80% do seu tempo.

Afeta todos os usuários ou apenas o departamento financeiro?
Afeta apenas a conexão Wi-Fi ou a rede cabeada também?
É um problema com o Servidor A ou com o Switch de Core?

Fase 03: Por que falhou? (RCA)

A Técnica dos "5 Porquês"

Aplicada à Gerência de Redes:

1. Por que o sistema parou? Porque o banco de dados não responde.
2. Por que o BD não responde? Porque perdeu conexão com a rede.
3. Por que perdeu conexão? Porque a porta 10 do switch está DOWN.
4. Por que a porta está DOWN? Porque o cabo de fibra rompeu.
5. Por que rompeu? Porque não havia proteção mecânica (eletroduto) em uma área de tráfego.

Encontramos a Causa Raiz!

Fase 04: Implementação e Correção

Regras de Ouro SRE

  • 1
    Uma mudança por vez: Nunca altere várias configurações simultaneamente.
  • 2
    Plano de Rollback: Saiba exatamente como desfazer o que você está prestes a fazer se der errado.
  • 3
    Comunicar Stakeholders: Avise quem será afetado pela interrupção momentânea da manutenção.

Fase 05: Documentar ou Repetir

Base de Conhecimento

Se você conserta algo e não documenta, você está condenando seu "eu do futuro" a gastar o mesmo tempo na próxima vez.

Estrutura Sugerida de Log de Resolução
  • • ID do Incidente: #1029
  • • Sintoma: Latência intermitente
  • • Causa: Loop de Camada 2 no Switch B
  • • Resolução: Ativação de Rapid STP
  • • Responsável: [Seu Nome]

Estratégias de Abordagem

Bottom-Up (Camada 1 -> 7)

"Primeiro vejo se o cabo está ligado e o link está aceso."

Ideal para problemas de conectividade total ou instabilidade elétrica/física.

Top-Down (Camada 7 -> 1)

"Primeiro vejo se a página web carrega ou se há erro HTTP."

Ideal quando apenas um serviço específico falha, enquanto o resto da rede está normal.

Logs: O DNA dos Eventos

Na **Gerência de Falhas (FCAPS)**, os logs são a nossa principal evidência. Um log é um registro cronológico de eventos gerado por um hardware ou software.

Logs Reativos

Gerados quando algo dá errado (ex: SSH login failed).

Logs Informativos

Apenas registros de estado (ex: Interface Gigabit 0/1 changed state to UP).

Anatomia de uma Linha de Log

# Exemplo de Syslog Linux
Feb 24 19:42:01 meta-srv-01 sshd[1245]: Failed password for root from 192.168.1.10
TimestampData/Hora
HostnameOrigem
Process/TagServiço
MessageO Evento

Logs de Ativos (Cisco)

Como os roteadores conversam com você:

%LINK-3-UPDOWN: Interface GigabitEthernet0/1, changed state to down
%LINEPROTO-5-UPDOWN: Line protocol on Interface GigabitEthernet0/1, changed state to down
%SYS-5-CONFIG_I: Configured from console by vty0 (10.1.1.5)
Expert Tip

O número central no código do log (ex: o '3' em LINK-3-UPDOWN) indica a severidade. 0 é Emergência, 7 é Debug. Memorize: quanto menor o número, mais você deve correr.

Outros Mundos: Windows & Linux

Event Viewer

No Windows, os logs são estruturados em arquivos .evtx (Application, Security, System).

Level: Error
Source: Service Control Manager
EventID: 7034
Message: The SQL Server service terminated unexpectedly.

Syslog / Journald

No Linux, quase tudo vai para /var/log/syslog ou é lido via journalctl.

$ journalctl -u nginx
-- Logs begin at Tue 2026...
Feb 24 19:45:01 server nginx[80]: [error] open() "/var/www/html/index.html" failed (2: No such file)

Caso: O Gargalo Misterioso

Sintoma Reportado

"O sistema de vendas da filial leste fica extremamente lento todos os dias, mas apenas no horário das 14h às 16h."

Isolamento Temporal: 14h - 16h diário.

Resolvendo o Gargalo...

1

Passo 1: Olhamos os gráficos de tráfego. Notamos que o link de 100Mbps chega a 99Mbps exatamente às 14:05.

2

Passo 2: Analisamos os logs do servidor. Descobrimos um processo de Backup automático agendado para rodar às 14h.

3

Passo 3: Causa Raiz? Gerência de Performance falhou ao agendar uma tarefa pesada em horário comercial.

Lição: Nem tudo que está lento é culpa do hardware. Às vezes, o software está apenas fazendo o que mandaram, na hora errada.

Atividade: Log Explorer

Coloque estes logs em ordem lógica de ocorrência:

B) 19:10:05 - Connection Refused on Port 80
A) 19:10:01 - Interface FastEthernet0/1 changed state to DOWN
C) 19:10:30 - User report: "Webpage not loading"
